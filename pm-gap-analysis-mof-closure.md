# 電子發票使用者任務閉環缺口分析

## 1. 文件背景與目的

本文件目的在於盤點現有 iOS / Android source code 與財政部電子發票整合服務平台之間的功能閉環差距。

本次盤點**不是單純比較功能有無**，而是從使用者任務角度出發，評估：

> 使用者在哪些情境下，無法在 App 內完成任務，仍需要回到財政部平台查詢、補設定或完成流程。

透過這份盤點，可協助產品、設計與開發團隊辨識：
- 哪些缺口最影響使用者體驗
- 哪些功能會造成使用者跳出 App
- 哪些任務應優先補足，以降低回流財政部的需求

---

## 2. 盤點範圍與方法

### 2.1 盤點範圍
本次盤點涵蓋：
- 財政部電子發票整合服務平台中，與手機條碼 / 載具 / 查詢 / 領獎 / 驗證相關功能
- iOS source code：`invoice-ios-readonly`
- Android source code：`invoice-android-readonly`

### 2.2 判斷原則
每個功能項目以以下問題判斷：
1. 原生 App 是否已有明確功能支援
2. 該功能是否能在 App 內完成閉環
3. 若無法閉環，使用者是否會被迫回到財政部平台
4. 該情境是否屬於高頻、核心、易流失任務

### 2.3 盤點標準
本文件依照「使用者是否需要回財政部」將缺口分為：
- **高優先缺口**：高頻且直接影響主線任務閉環
- **中優先缺口**：非所有人都會遇到，但與主線高度相關
- **低優先缺口**：偏官方查詢 / 行政資訊型功能，非一般日常主線

---

## 3. 核心結論摘要

依照本次盤點，最需要優先補足的缺口為：

1. **查詢載具發票**
2. **忘記驗證碼 / 驗證碼異常處理**
3. **領獎設定 / 銀行帳戶設定**

這三項的共通點是：
- 都屬於高頻或高關鍵任務
- 一旦缺失，使用者容易中斷當前任務
- 使用者會自然回到財政部平台確認或補完成
- 對使用者信任感與體驗完整性影響最大

---

## 4. 高優先缺口分析

### 4.1 查詢載具發票

#### 使用者情境
- 明明有消費，但 App 裡找不到某張發票
- 剛完成歸戶，想確認發票是否真的進來
- 想核對 App 顯示與官方資料是否一致
- 換手機 / 重新登入後，想確認官方帳上有哪些發票

#### 為什麼會回財政部
財政部平台是官方發票資料源。只要 App 內資料顯示不完整、同步延遲、歸戶關係不透明，使用者就會回到財政部查詢載具發票。

#### source code 盤點結果
原生 App 雖已有：
- 發票查詢 / 搜尋能力
- 載具管理
- 歸戶功能

但目前從盤點邏輯來看，**仍缺少一個清楚的「官方載具發票查詢 / 對照入口」來承接使用者 fallback 行為**。

#### 對使用者影響
- 找不到發票時容易懷疑 App 不準
- 對歸戶結果缺乏信任
- 發票查詢不完整時，會直接轉向官方平台

#### 產品判斷
這是**高頻且自然發生的 fallback 行為**，應視為最高優先缺口。

#### 建議方向
- 建立 App 內的「查詢載具發票」入口
- 提供查無發票時的可能原因說明
- 區分「未歸戶」「同步中」「查無官方資料」等狀態
- 若需 fallback 至官方，也應在 App 內有更清楚承接

---

### 4.2 忘記驗證碼 / 驗證碼異常處理

#### 使用者情境
- 忘記載具驗證碼
- 載具登入失敗
- 驗證碼過期 / 錯誤
- 載具雖存在，但無法通過登入驗證

#### source code 線索
Android 明確有：
- `forgotPasswordUrl`
- `MofWebLoginActivity`
- `NavigateMofLogin`
- `UpdateCarrierPasswordScreen`

iOS / Android 也都有密碼 / 驗證碼更新相關流程。

#### 為什麼會回財政部
App 雖然有處理入口，但從 source code 可見，某些 case 仍依賴：
- MOF web fallback
- forgot password URL
- 官方登入補流程

也就是說，**這條任務並未完全在 App 內閉環**。

#### 對使用者影響
- 一旦登入卡住，使用者無法使用後續所有功能
- 若必須切回官方頁面，流程斷裂感強
- 驗證碼問題通常發生在使用者本來就焦慮的情境下，容錯要求更高

#### 產品判斷
這是**登入閉環的核心缺口**，重要性極高。

#### 建議方向
- 在 App 內提供更明確的驗證碼異常處理路徑
- 優化忘記驗證碼入口與文案
- 區分「驗證碼忘記」「驗證碼錯誤」「帳號尚未完成註冊」等情境
- 若仍需官方 fallback，應降低跳轉感與中斷感

---

### 4.3 領獎設定 / 銀行帳戶設定

#### 使用者情境
- 第一次設定中獎匯款帳戶
- 修改領獎設定
- 想確認自己是否已完成設定
- 中獎後才發現尚未綁定領獎資料

#### source code 線索
iOS 有：
- `SetBankAccountInfo.swift`
- `GetSetBankAccountInfoRequest.swift`
- `IBSettingWebViewVC.swift`

Android 有：
- `bankAccountStatus`

#### 為什麼會回財政部
目前判斷這塊較像：
- App 可查詢狀態
- 但實際設定流程可能仍依賴 web / 財政部頁面

因此，**領獎設定尚未形成完全原生閉環**。

#### 對使用者影響
- 中獎時是高情緒、高期待時刻
- 若這時才發現要跳回官方設定，體驗落差最大
- 容易造成「App 可以查中獎，但真正領獎還要自己補做」的割裂感

#### 產品判斷
這是**關鍵閉環缺口**，因為它直接影響中獎後的最終價值兌現。

#### 建議方向
- App 內清楚顯示是否已完成領獎設定
- 將銀行帳戶設定前置化，不要等中獎才補做
- 若仍需 web 流程，應清楚標示目前狀態與下一步
- 可增加「未設定提醒」「設定完成確認」等 UX

---

## 5. 中優先缺口分析

### 5.1 查詢已綁定載具 / 歸戶狀態

#### 使用者情境
- 不知道自己綁了哪些載具
- 不確定會員卡是否已歸戶成功
- 不知道某張發票落在哪個載具下面

#### source code 線索
原生 App 已有：
- `CarrierManagement`
- `MemberCarrier`
- `CarrierAndCards`
- `GetSubCarriersRequest`

#### 判斷
App 已具備一定管理能力，但若資訊呈現不夠透明，使用者仍會回財政部確認官方載具關係。

#### 產品建議
- 強化載具狀態透明度
- 清楚顯示每個載具的歸戶狀態
- 提供「這張發票來自哪個載具」的說明

---

### 5.2 自然人憑證相關流程

#### 使用者情境
- 不想使用手機條碼 / 驗證碼路徑
- 需要更正式的官方身份驗證方式
- 某些官方功能可能要求使用自然人憑證

#### 判斷
本次盤點未發現明確原生 module 對應，因此若使用者需走此路徑，高機率仍需回財政部。

#### 產品建議
- 若此功能目標使用者有限，可暫列中優先
- 至少應在 App 中說明替代路徑與入口位置

---

## 6. 低優先缺口分析

### 6.1 電子發票營業人查詢
#### 使用者情境
- 想確認某商家是否為電子發票營業人
- 查詢某通路是否支援電子發票

#### 判斷
屬於官方資訊查詢型功能，非一般使用者高頻主線。

### 6.2 加值服務中心查詢
#### 使用者情境
- 偏業者 / 系統服務查詢
- 一般消費者很少主動使用

#### 判斷
非核心消費者流程，優先度低。

### 6.3 全民稽核專區
#### 使用者情境
- 特定情境才會使用
- 不屬於一般日常消費主線

#### 判斷
低優先。

### 6.4 線上申請（泛行政型）
#### 使用者情境
- 特殊行政申辦
- 補件 / 特定流程

#### 判斷
偏官方行政入口，非本產品主線。

---

## 7. 優先順序建議

### Phase 1：優先補強閉環
1. 查詢載具發票
2. 忘記驗證碼 / 驗證碼異常處理
3. 領獎設定 / 銀行帳戶設定

### Phase 2：補強透明度與管理能力
4. 查詢已綁定載具 / 歸戶狀態
5. 自然人憑證相關流程評估

### Phase 3：補足官方查詢型服務
6. 電子發票營業人查詢
7. 加值服務中心查詢
8. 全民稽核專區
9. 線上申請

---

## 8. 對產品的意義

本次盤點顯示，原生 App 並不是完全缺少財政部相關功能，而是：

- **主線功能大致存在**
- 但在少數關鍵任務上，仍未完全閉環
- 這些缺口一旦發生，會直接把使用者推回財政部平台

因此，真正的產品問題不是「功能數量不夠」，而是：

> **在使用者最需要完成任務的時刻，App 是否能自己承接到底。**

高優先缺口共同指向的核心課題是：
- 信任
- 閉環
- fallback 承接能力

---

## 9. 結論

若以「使用者是否需要回財政部」作為判準，最關鍵的缺口為：

1. **查詢載具發票**
2. **忘記驗證碼 / 驗證碼異常**
3. **領獎設定 / 銀行帳戶設定**

這三項優先補齊後，才能有效降低：
- 使用者對官方平台的依賴
- 任務中斷率
- 對 App 資料可信度的懷疑

也才能讓 App 從「功能入口」進一步升級為真正可閉環完成任務的使用體驗。
